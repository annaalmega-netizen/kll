# Влияние генеративного ИИ на сохранность знаний студентов

Учебный проект (Вариант 3): полный воспроизводимый пайплайн машинного обучения
вокруг вопроса **«Влияет ли чрезмерное использование ИИ на долгосрочную
сохранность знаний?»**

Данные: `data/ai_student_impact_dataset.csv` — 50 000 студентов, 16 признаков,
без пропусков. Целевая переменная ML-задачи (регрессия) — `Skill_Retention_Score`.

## Задание 1 — EDA

```bash
python eda/run_eda.py
```

Строит 5 графиков разного формата (гистограмма+KDE, диаграмма рассеяния с
трендом, boxplot, heatmap корреляций, barplot) в `eda/figures/` и печатает
статистические выводы. Ключевые находки:

1. **Часы ИИ ↔ сохранность отрицательно связаны** (Пирсон r=−0.118, p≈1e‑154).
2. **Тяжёлые пользователи** (верхние 25%, ≥11.7 ч/нед) имеют сохранность 74.0
   против 76.4 у остальных (t=−16.6, p≈1e‑61).
3. **Сценарий использования значим** (ANOVA F=141, p≈1e‑120): пассивное
   «получение готовых ответов» — хуже всего (73.7), активная «отладка» —
   лучше всего (78.1).
4. **Традиционная учёба** — защитный фактор (r=+0.148).

Вывод: да, чрезмерное и *пассивное* использование ИИ ассоциировано с худшей
сохранностью знаний; традиционная учёба частично компенсирует эффект.

## Задание 2 — ML-задача

**Регрессия**: предсказание `Skill_Retention_Score`. Обучаются ≥2 модели
(линейная регрессия и случайный лес), сравнение по RMSE:

| Модель         | RMSE  | MAE  | R²   |
|----------------|-------|------|------|
| Linear         | 11.96 | 9.68 | 0.188 |
| Random Forest  | 11.89 | 9.60 | 0.198 |

**Лучшая — Random Forest** (минимальный RMSE): улавливает нелинейности и
взаимодействия признаков, но выигрыш у линейной модели небольшой, т.к. целевая
переменная сильно зашумлена.

Интерпретация и анализ ошибок:

```bash
python eda/interpret_model.py
```

Наиболее влиятельные признаки (permutation importance): `Prompt_Engineering_Skill`,
`Primary_Use_Case`, `Weekly_GenAI_Hours`, `Traditional_Study_Hours` — согласуется
с EDA. Остатки почти несмещённые (bias≈0), MAE стабильна по квартилям часов ИИ.

## Задание 4 — паттерны проектирования

Применены **два паттерна из разных групп** (`src/genai_retention/models.py`):

- **Factory Method (порождающий)** — `ModelFactory.create(name)` инкапсулирует
  создание модели по строковому ключу. Проблема: пайплайн не должен знать
  конкретные классы моделей и менять код при добавлении новой. Решение: новая
  модель регистрируется в фабрике, остальной код не трогаем.
- **Strategy (поведенческий)** — `ModelStrategy` задаёт единый интерфейс
  (`build`) для взаимозаменяемых алгоритмов. Проблема: нужно сравнивать разные
  алгоритмы единообразно. Решение: `Trainer` работает с абстракцией и
  подменяет стратегию без изменения логики обучения.

Паттерны уместны, потому что задание прямо требует обучать и сравнивать
несколько моделей — фабрика и стратегия делают набор моделей открытым для
расширения и закрытым для изменения (принцип OCP).

## Задание 5 — консольное приложение

Использует сохранённую модель и код пайплайна:

```bash
# по одному студенту (флаги; недостающие поля = значения по умолчанию)
python app/cli.py --weekly-genai-hours 35 --traditional-study-hours 3 \
    --tool-diversity 1 --use-case Direct_Answer_Generation --skill Beginner \
    --dependency 9 --burnout High

# пакетно по CSV
python app/cli.py --input new_students.csv

# интерактивный режим
python app/cli.py --interactive
```

Приложение загружает `artifacts/model.joblib`, применяет тот же
`FeatureEngineer`, что и при обучении, и возвращает предсказанный
`Skill_Retention_Score`.
